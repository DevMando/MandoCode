# Changelog — March 20, 2026

Comprehensive session covering bug fixes, performance optimizations, UX improvements, new kernel functions, and technical debt resolution across the MandoCode codebase.

## Bug Fixes

### 1. Model Validation Recovery — App No Longer Freezes on Missing Model

**Files:** `Components/App.razor`, `Services/AIService.cs`

**Problem:** When the configured model wasn't found in Ollama (e.g., a typo or unpulled model), the app displayed a warning panel but never started the interactive loop. The user was stuck with no way to type, configure, or exit — the application was effectively bricked.

**Fix:** The interactive loop now always starts, even when model validation fails. A `_modelError` flag tracks degraded mode. In this mode:
- `/config`, `/help`, `/exit`, `/learn`, and `!shell` commands all work normally
- AI requests are blocked with a helpful message: "No model loaded. Run /config to select a model."
- After running `/config`, the model is re-validated. If valid, full mode activates immediately with a success message.
- The error panel shows actionable guidance: pull the model, run `/config`, or start `ollama serve`.

### 2. Redundant Project Root in File Paths — "File Not Found" During Plans

**File:** `Plugins/FileSystemPlugin.cs`

**Problem:** During task plan execution, the model sometimes included the project root in file paths (e.g., `src/MandoCode/bin/Debug/net8.0/Games/script.js` when the project root was already `src/MandoCode/bin/Debug/net8.0`). `Path.Combine` doubled up the path, causing "File not found" errors even though the file existed.

**Fix:** Added `StripRedundantRootPrefix()` in `GetFullPath()`. It detects when a relative path starts with any suffix of the project root and strips the redundant prefix. For example, if the model passes `src/MandoCode/bin/Debug/net8.0/Games/file.js`, it normalizes to `Games/file.js`. Also added explicit FILE PATH RULES to the system prompt telling the model to never include the project root in paths.

### 3. Spinner Not Visible After Function Calls — App Appears Frozen

**File:** `Components/App.razor`

**Problem:** After the model finished reading files or executing operations, the spinner restarted with no message (`Spinner.Start()` with no argument). The bare spinner animation was nearly invisible in many terminals, making the app look frozen while the model was thinking.

**Fix:** Added contextual spinner messages throughout:
- Initial request: `"Thinking..."`
- After reading 1 file: `"Analyzing file..."`
- After reading N files: `"Analyzing N files..."`
- After write/update: `"Applying changes..."`
- After search: `"Reviewing search results..."`
- After glob: `"Scanning project structure..."`
- After web search/fetch: `"Processing search results..."` / `"Reading web content..."`
- Plan step start: `"Working..."`

Added operation tracking fields (`_recentReadCount`, `_recentReadFiles`, `_lastOperationType`) that reset per request and accumulate during function execution.

### 4. No Spinner During Plan Step Execution

**File:** `Components/App.razor`

**Problem:** When a task plan step started, the step header printed but no spinner was shown. The user saw the step title then silence until function calls began.

**Fix:** Added `Spinner.Start("Working...")` immediately after the step header renders in `HandleProgressEventAsync.StepStarted`. Also reset the operation tracking fields per step.

### 5. Startup Shows /learn Content Flash

**Files:** `Components/App.razor`

**Problem:** Moving the Ollama connection check to `OnAfterRenderAsync` (for async improvement) caused the first render to see `_isConnected = false` and display `LearnContent.Display()` before the check completed.

**Fix:** Reverted the connection check back to `OnInitialized` (synchronous). The 5-second timeout HTTP GET is fast enough that async wasn't worth the rendering complexity. Added `_connectionChecked` flag for future use.

## Performance Optimizations

### 6. Completion Tracker Wait Reduced from 30s to 5s

**Files:** `Services/AIService.cs`, `Services/TaskPlannerService.cs`, `Services/FunctionCompletionTracker.cs`

**Problem:** After every plan step and AI operation, `WaitForAllCompletionsAsync(TimeSpan.FromSeconds(30))` blocked for up to 30 seconds even when all functions had already completed. The semaphore-based tracker has a fast path (returns immediately when `_pendingCount == 0`), but the timeout was hit whenever the signal wasn't perfectly synchronized.

**Fix:** Reduced all wait points from 30s to 5s. Functions almost always complete before the AI response returns, so the 30s timeout was pure waste. For a 5-step plan, this saves up to 125 seconds of dead wait time.

### 7. History Lock Narrowed — No Longer Held During AI Calls

**File:** `Services/AIService.cs`

**Problem:** The `_historyLock` semaphore was held for the entire duration of `GetChatMessageContentAsync()` (up to 5 minutes) in both `ChatAsync` and `ChatStreamAsync`. This was a critical anti-pattern — the lock only needed to protect the quick add/remove message operations.

**Fix:** Lock is now acquired only for the `AddUserMessage` and `AddAssistantMessage` calls (microseconds), then released before the long AI operation. Eliminates lock contention entirely.

### 8. Concurrent Function Execution Enabled in Plan Steps

**File:** `Services/AIService.cs`

**Problem:** `ExecutePlanStepAsync` set `AllowConcurrentInvocation = false`, forcing all function calls within a step to execute sequentially. If a step needed to read 3 files, they ran one at a time.

**Fix:** Changed to `AllowConcurrentInvocation = true`, matching the setting used for direct requests. Parallel file reads within a single step now execute concurrently.

### 9. Previous Step Results Capped to Last 2

**File:** `Services/AIService.cs`

**Problem:** `ExecutePlanStepAsync` included ALL previous step results as context in the prompt. Step 8 included results from steps 1-7, causing quadratic token growth. Later steps sent increasingly large prompts, slowing model responses.

**Fix:** Only the last 2 step results are included as context. Step 8 sees results from steps 6-7 only. Keeps prompts small and response times consistent across long plans.

### 10. Plan Steps Excluded from Main Chat History

**File:** `Services/AIService.cs`

**Problem:** Every plan step was appended to the main `_chatHistory` (user message + assistant response). A 10-step plan added ~20 messages to the history, bloating all subsequent requests even after the plan completed.

**Fix:** Removed the code that appended plan steps to `_chatHistory`. Plan steps use their own isolated `ChatHistory` per step. Post-plan requests start clean.

### 11. File List Caching with 5-Second TTL

**File:** `Plugins/FileSystemPlugin.cs`

**Problem:** Every call to `ListAllProjectFiles`, `ListFiles`, `FindInFiles`, or `GrepFiles` triggered a full recursive directory walk of the project. On large projects, this was expensive and happened multiple times per AI request.

**Fix:** Added a file list cache (`_cachedFiles`) with a 5-second TTL. All read operations use `GetAllFilesCached()`. Cache is invalidated immediately on any write, edit, delete, or create operation via `InvalidateCache()`.

## New Kernel Functions

### 12. `edit_file` — Targeted Find/Replace

**Files:** `Plugins/FileSystemPlugin.cs`, `Services/FunctionInvocationFilter.cs`, `Models/SystemPrompts.cs`

**Why:** The model previously had to read an entire file, modify it mentally, and write the whole thing back with `write_file`. For a one-line change in a 500-line file, this wasted tokens and produced noisy diffs.

**Implementation:**
- `edit_file(relativePath, old_text, new_text)` — finds exact text match and replaces it
- Validates exact match (returns error if not found)
- Rejects ambiguous multi-match (returns error if text appears more than once)
- Goes through the full diff approval flow — constructs the new file content from the replacement and shows a proper diff panel
- Wired into `FunctionInvocationFilter` for operation display, deduplication, and approval interception
- System prompt updated to guide the model to prefer `edit_file` over `write_file` for existing file modifications

### 13. `grep_files` — Search All Project Files

**Files:** `Plugins/FileSystemPlugin.cs`, `Services/FunctionInvocationFilter.cs`

**Why:** The existing `search_text_in_files` required a glob pattern, which models often got wrong. A simple "search everything" function is what they actually need most of the time. Also, `FindInFiles` loaded entire files into memory.

**Implementation:**
- `grep_files(searchText)` — searches all project files for text, case-insensitive
- Streams line-by-line with `StreamReader` (no full file load into memory)
- Results capped at 50 matches to prevent unbounded output
- Returns file paths with line numbers and matching line content
- Wired into `FunctionInvocationFilter` with "Search" operation display

## Technical Debt Resolution (CodeImprovements.md)

### 21. (#1) Blocking Async — Analyzed and Kept

**File:** `Components/App.razor`

The `CheckOllamaConnectionAsync().GetAwaiter().GetResult()` in `OnInitialized` was attempted to be moved to `OnAfterRenderAsync` but caused rendering issues (learn content flash, missing HelpDisplay). Reverted — the synchronous call with a 5-second timeout is acceptable for startup.

### 22. (#3) Duplicate Ignore Directory Lists — Centralized

**Files:** `Models/MandoCodeConfig.cs`, `Plugins/FileSystemPlugin.cs`, `Program.cs`, `Services/ConfigurationWizard.cs`

Added `MandoCodeConfig.DefaultIgnoreDirectories` static field. All 4 locations (FileSystemPlugin, Program.cs DI, CreateDefault(), ConfigurationWizard reset) now reference it. Added `.claude` to the canonical list.

### 23. (#4) Shell Command OOM Risk — Streaming with Cap

**File:** `Services/ShellCommandHandler.cs`

Replaced `ReadToEnd()` with line-by-line streaming via `ReadLine()`. Output capped at 100,000 characters with truncation notice. Output appears incrementally as the command runs instead of waiting for completion.

### 24. (#5) Duplicate JSON Brace-Counting Logic — Shared Helper

**File:** `Services/AIService.cs`

Extracted `FindJsonObjectBounds(string text, int startIndex)` that returns `(int Start, int End)?`. Both `ExtractJsonObject` and `RemoveFunctionCallJson` now call this shared helper instead of duplicating the brace-depth/string-escape tracking logic.

### 25. (#6) No File List Caching — 5-Second TTL Cache

**File:** `Plugins/FileSystemPlugin.cs`

See item #11 above.

### 26. (#7) Awkward String Construction — Removed Entirely

**File:** `Plugins/FileSystemPlugin.cs`

The `$"{'='.ToString().PadRight(50, '=')}\n"` separator line was removed entirely as part of the ReadFile cleanup (item #14).

### 27. (#8) Missing IDisposable on App.razor

**File:** `Components/App.razor`

Added `@implements IDisposable` and `Dispose()` method that cleans up: spinner, music visualizer, request CTS, music player, and theme service. Prevents resource leaks on unexpected exit.

### 28. (#9) Process Exit Code Not Checked

**File:** `Services/ShellCommandHandler.cs`

`WaitForExit(30_000)` return value is now checked. If the process doesn't exit within 30 seconds, it's killed and the user sees "Command timed out after 30 seconds."

### 29. (#10) FindInFiles Reads All Files Into Memory

**File:** `Plugins/FileSystemPlugin.cs`

Replaced `File.ReadAllLinesAsync` with `StreamReader` line-by-line streaming. Results capped at 50 matches. See item #18.

## Earlier Bug Fixes (from codebase audit)

### 30. LoopStream Infinite Loop Guard

**File:** `Services/MusicPlayerService.cs`

Added `loopResets` counter to `LoopStream.Read()`. Breaks after 3 consecutive zero-read resets to prevent infinite loop on corrupted/empty MP3 streams.

### 31. Race Condition in Plan Status

**File:** `Services/TaskPlannerService.cs`

Added `_planStatusLock` protecting `plan.Status` reads in `ExecutePlanAsync` and writes in `CancelPlan`. Prevents user cancellation from being missed due to concurrent access.

### 32. Spinner Cleanup Try/Finally

**File:** `Services/SpinnerService.cs`

Wrapped `cts.Cancel()` + `task.Wait()` in try/finally so `cts.Dispose()` always runs even if an exception occurs.

### 33. Symlink Cycle Protection

**File:** `Plugins/FileSystemPlugin.cs`

`GetAllFiles()` now tracks visited directories via `HashSet<string>`. Skips already-seen paths to prevent `StackOverflowException` from symlink cycles.

### 34. ReDoS Prevention in Task Plan Regex

**File:** `Services/TaskPlannerService.cs`

Replaced `\s*[-:]\s*` with `[\s]*[-:][\s]*` in character class split pattern to prevent catastrophic backtracking.

### 35. Stream Leak in Music Track Discovery

**File:** `Services/MusicPlayerService.cs`

Changed embedded resource stream to `using var` so it's properly disposed even if `CopyTo()` throws.

### 36. Bounds Check in `@` Autocomplete Trigger

**File:** `Services/CommandAutocomplete.cs`

Added `cursorPos >= 2` guard before accessing `input[cursorPos - 2]` to prevent `IndexOutOfRangeException` when `@` is typed at position 1.

### 37. ReDoS Prevention in URL Detection

**File:** `Services/MarkdownRenderer.cs`

Added length limits (`{1,2000}`, `{0,250}`) to greedy quantifiers in `UrlPattern` and a 2-second regex timeout.

### 38. LoopStream Position Seekable Guard

**File:** `Services/MusicPlayerService.cs`

`Position` setter now validates `CanSeek` before assignment, throwing `NotSupportedException` for non-seekable streams.

### 39. Better Large-File Diff Fallback

**File:** `Services/DiffService.cs`

When LCS table exceeds 25M cells, now shows first/last 50 lines from each side with an omission marker instead of dumping all lines as removed + all lines as added.

### 40. Compiled Whitespace Regex in WebSearchPlugin

**File:** `Plugins/WebSearchPlugin.cs`

Replaced per-line `Regex.Replace(line, @"[ \t]+", " ")` with pre-compiled static `InlineWhitespacePattern` regex.
